# ButFun 3D — 直播優先的 3D 多人遊戲（設計藍圖）

> 狀態：規劃中（2026-06-26 起）。本文件是「平行第二軌」的藍圖。
> 第一軌（2D ButFun 自走迴圈）持續完善試驗場，**不受本專案影響**。

## 0. 一句話

**拿被 2D ButFun 驗證過的系統，做一個「瀏覽器即玩、為直播而生」的 3D 多人療癒遊戲。**

---

## 1. 策略定位（為什麼、怎麼跟 2D 並存）

- **2D ButFun ＝ 先行試驗場／設計藍本（proving ground）**：便宜、快、瀏覽器即玩，用來把玩法／系統／平衡／樂趣**測到好、磨到完善**。由自走迴圈持續推進。
- **本專案（3D）＝ 平行第二軌**：拿 2D 驗證過的系統，做一個新的 3D 遊戲。
- **兩軌不互卡**：2D 迴圈自動完善；3D 由人＋AI 平行架。
- **「依樣畫葫蘆」＋ 一個進化**：3D 重用 2D 的底層系統與世界，但加上**「直播優先的多人玩法」**這個 2D 沒有的新維度——這是本作的差異化與爆發點。
- **最值錢的可重用資產**：ButFun 的邏輯與多人後端都是 **Rust**（`world-core` 地形/移動、`ws.rs` WebSocket 多人）。新 3D 直接站在這個肩膀上，省掉最難的部分。

---

## 2. 引擎決策：瀏覽器 3D（**不是**原生下載）

**結論：Three.js（WebGL）瀏覽器 3D。**

- **理由＝直播**：直播爆紅的命脈是**「一鍵入場」**——主播喊網址、觀眾秒進。**瀏覽器即玩＝病毒擴散超能力**；下載版（安裝＋商店審核＋帳號牆）會把這條病毒迴圈直接掐死。參考 `.io` 類遊戲（agar.io 等）＝瀏覽器＋即玩＋天生好播＋病毒擴散。
- **技術**：Three.js 跑 **low-poly 程式生 mesh**（體驗 > 精緻，沿用 2D 的結論：[[art-direction-owned-by-ai]] 黏土微縮風）。手機（iPhone Safari）/桌面都跑得動。
- **重用 ButFun 現有 Rust 多人後端**（`ws.rs` 的 WebSocket、`game.rs` 的快照廣播）＋ `world-core`（地形/移動）——**最難的「多人連線」已經有了**。
- **原生／Steam 版**：等市場驗證「這遊戲會紅」後再考慮（另一條人力管線）。現在**不做**。

---

## 3. 北極星：直播優先（the streaming flag）

**這是本作的核心差異化。每個設計決策都要過一遍濾鏡：「這對直播好嗎？」**

**設計支柱（烤進每一局）：**
1. **好看好剪的時刻**：社交、混亂、搞笑、逆轉——每局都該產生值得剪成短影音的瞬間。
2. **觀眾一眼看懂**：清晰的視覺語言、旁觀視角（spectator cam），不用解說也知道在幹嘛。
3. **低門檻一鍵入場**：網址／房間碼，無安裝、無帳號牆（訪客即玩優先，登入可選）。
4. **Streamer mode**：遮個資、**防 stream-snipe**（對手拿不到主播畫面上的資訊、或加延遲），伺服器端強制。
5. **觀眾互動 hook**：觀眾投票觸發事件／丟道具／加入當搗蛋角色——把觀眾變成玩法的一部分。
6. **短局、易上手、難精通**：適合一場一場剪、一場一場約。

**旗艦模式：3D 療癒版「變裝躲貓貓」（prop-hunt / hide-and-seek）**
- 躲方**變裝成世界中的物件**（南瓜、木桶、蘑菇…黏土風），找方要把他們抓出來。
- **天生好播**：找方走過偽裝物的緊張、躲方憋笑、抓錯的尷尬搞笑、最後一刻的逆轉。
- **搭 ButFun 療癒社交調性**：不血腥、可愛、社交——是「療癒版」躲貓貓，不是恐怖追逐。
- **觀眾 hook**：觀眾可投票給找方一個提示、或幫躲方放一陣煙霧。
- 變裝躲藏是**被市場驗證過的直播金礦類型**（變色龍躲貓貓、prop hunt 一票），但**我們寫原創實作**（只借通用玩法概念，守 [[game-design-references-legal]] 法律紅線，絕不抄碼）。

---

## 4. 重用 vs 新做

**重用（從 ButFun，這是省力的關鍵）：**
- `world-core`（Rust→wasm）：地形生成、移動物理（[[wasm-terrain-single-impl]] 單一實作）。
- `ws.rs` / `game.rs`：WebSocket 多人連線、快照廣播骨架、伺服器權威移動。
- 視模式需要選用的遊戲系統（採集／合成／經濟／作物…）。
- 美術北極星（黏土微縮）＋生圖管線（deb-pc，[[image-gen-pipeline-deb-pc]]）。

**新做（本專案）：**
- **Three.js 3D 前端**：場景／透視鏡頭／mesh／光影——程式生 low-poly。
- **房間／對局制**：lobby、房間碼、短局賽制。ButFun 是「持續世界」，躲貓貓是「對局制」→ 需要新的**房間／對局層**。
- **躲貓貓模式邏輯**：變裝、抓捕、計分、勝負、回合。
- **直播層**：streamer mode（資訊隱藏／防 snipe）、觀眾互動、旁觀視角、分享／剪輯 hook。

---

## 5. 第一個 spike（最小可玩，先驗證「這條路通」）

**目標：證明「瀏覽器 3D ＋ 重用後端多人」可行。** 不求完整、不求精緻。

- Three.js 3D 場景：low-poly 地面（從 `world-core` 地形）＋ 透視鏡頭。
- 多人：連上 ButFun 既有 WebSocket（或一個精簡分支），**在 3D 世界裡看到別的玩家在跑**。
- 極簡躲貓貓雛形：能變裝成一個物件、有人能來抓、抓到有反饋。
- **驗收**：兩支手機／瀏覽器開同一網址，在 3D 世界裡互相看得到、玩一局最陽春的躲貓貓。

---

## 6. 階段（每階段都過「直播好嗎」濾鏡）

- **P0 — Spike**（上一節）：證明可行。
- **P1 — 旗艦躲貓貓 MVP**：完整一局（lobby → 變裝躲藏 → 抓捕 → 計分 → 結算），黏土 3D，手機可玩。
- **P2 — 直播層**：streamer mode、觀眾互動、旁觀視角、一鍵分享／剪輯 hook、房間碼擴散。
- **P3 — 打磨／擴充**：更多地圖、更多變裝、更多模式、輕進程感、回流鉤子。

---

## 7. 架構決策與開放問題

- **同 repo 還是新 repo？** 建議：**先在 ButFun repo 開 `web/3d/`（或 `experiments/3d/`）做 spike、重用後端**；證明可行後再決定是否拆獨立 repo（避免太早分家、增加維護成本）。
- **對局制 vs 持續世界**：躲貓貓是對局制（房間／短局），跟 ButFun 持續世界不同 → 要加**房間／對局層**（可加在現有 server，或獨立 game-mode 服務）。spike 階段先走最小路徑（純重用 `world-core` 地形/移動 + 一個極簡房間）。
- **多人權威**：維持**伺服器權威**（守 ButFun 好架構，別把判定放前端）。
- **反作弊／防 snipe**：streamer mode 的資訊隱藏**必須在伺服器端**做。
- **帳號**：直播低門檻 → **訪客即玩優先**，登入可選。
- **變現**：先不管，**先做出「會紅的體驗」**再說。
- **能不能跟 2D 共用世界？** 短期：否（對局制 vs 持續世界不同）。長期：3D 躲貓貓可發生在「ButFun 世界的一角」，做世界觀連結。

---

## 8. 不做什麼（範圍紀律）

- **不做原生下載版**（現在）——會掐死直播的一鍵入場。
- **不在 spike 階段追求精緻 3D**（體驗 > 精緻）。
- **不影響 2D 迴圈**——它繼續完善試驗場（第一軌）。
- **不抄外部遊戲碼**——躲貓貓／prop-hunt 是通用玩法概念，**原創實作**，守 [[game-design-references-legal]] 法律紅線。
- **不一次做完**——先 spike 驗證可行，再逐階段長。
