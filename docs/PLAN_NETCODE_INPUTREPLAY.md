# 正規 netcode：輸入重放對帳（input-replay reconciliation）

> 目標：把 ButFun 3D（與可選 2D）的移動從「朝權威慢慢靠」的**近似法**，升級成 FPS/LoL 等級的
> **輸入重放對帳**——真正消掉殘留抖動（不是只降 55%）。維護者拍板「要做就做好」。
> **何時做**：Claude 額度恢復後（明晚 10 點自動回原狀＝agy 免費 worker＋Sonnet＋80%，正好用免費 Gemini 做）。
> **這是迴圈恢復後的最高優先（先於其他 AI 世界切片）。**

## 為什麼現行版會殘留抖動
現在客戶端「每幀朝最新權威位置平滑收斂」。但權威位置：①比輸入慢一個 RTT、②在 prod 是**叢發抵達**（1–144ms）。所以收斂目標忽快忽慢 → 殘留微抖。外插只能緩解、消不掉。

## 正解：輸入重放（input replay）
每個輸入編號；伺服器回報「我處理到第幾號」；客戶端**從權威位置把『還沒被確認的輸入』重跑一遍** → 預測精準重建、零抖動、零漂移。FPS/LoL 都這麼做。

## 後端改動（Rust，向後相容）
1. **`ClientMsg::Input` 加 `seq: u32`**（`#[serde(default)]`）——客戶端每送一次輸入遞增。
2. **`Player` 加 `last_input_seq: u32`**（`state.rs`）——`ws.rs` 處理該玩家輸入時記下收到的 seq。
3. **自身快照回傳 `last_input_seq`**（`protocol.rs` 的玩家 view 加欄位 `#[serde(default)]`；`game.rs` snapshot 帶上）——讓客戶端知道「伺服器處理到第幾號」。
4. 守鎖紀律（不在持鎖中 await）、向後相容（舊客戶端不送 seq＝0、行為退回現況）。

## 前端改動（`web/3d/main.js`；同法可選擇套 `web/game.js`）
1. 每次送輸入 `seq++`，把 `{seq, mask, dt}` 推進一個**輸入歷史緩衝**。
2. 收到自身快照（含權威座標 + `last_input_seq`）時：
   - 預測位置 = **權威座標**（基準）。
   - 把歷史裡 `seq > last_input_seq` 的輸入（伺服器還沒確認的）用 **wasm `step_player`** 從基準**重跑一遍**。
   - 結果 = 從正確基準重建的預測，**不抖、不漂**（不再朝移動目標拔河）。
   - 丟掉已確認（`seq <= last_input_seq`）的歷史。
3. 每幀照常往前預測新輸入。大跳躍（傳送）仍直接 snap。
4. wasm 失敗時退回現行收斂法（保命）。

## 量測（用既有 QA 工具）
- 改完用 `scripts/qa/move-qa.mjs`（擴充成模擬「輸入重放」）對 prod 實測：**殘留單幀 lurch 應降到 ≈0**、誤差仍是健康的 ~1 tick 領先。
- **先量再宣稱修好**（沿用「不再瞎猜」的紀律）。
