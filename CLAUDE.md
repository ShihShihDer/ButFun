# ButFun — 給 AI 開發者的工作守則

ButFun 是一個瀏覽器即玩的**蒸汽龐克太空歌劇療癒多人世界**。
完整願景見 `docs/GAME_DESIGN.md`，待辦順序見 `docs/BACKLOG.md`。

這個專案以「**收建議 → 改版 → 發佈**」的迴圈持續長大。你（AI）會被反覆叫來推進它，
常常是透過 `/goal` 指令自走。以下是讓你能放手做、又不脫軌的**邊界**。

## 北極星 vs 紀律（最重要）

- 願景是多星球星際宇宙，但那是**長期方向，不是現在**。
- **一次只推進一個垂直切片**，照 `docs/BACKLOG.md` 由上往下做。
- **不要跳級**：Phase 0 沒穩之前，不碰飛船 / 多星球 / 競速。
- 寧可小而完整，不要大而半成品。範圍蔓延是這專案的頭號殺手。

## 品質閘門（push 前必過）

1. `cargo build` 必須成功。
2. `cargo test` 必須全綠。新功能要附上對應測試（純邏輯抽成可測函式）。
3. 伺服器要能實際啟動（`cargo run` 不 panic）。
4. 不要 push 壞掉的 build。測試失敗就照實說、修好再 push。

## 程式風格

- 後端 Rust（axum + tokio）；前端原生 HTML5 Canvas（暫不引入框架，要引入先記在 BACKLOG）。
- 跟著現有檔案的風格、命名、註解密度走。註解用繁體中文。
- 持久化一律走 `state.rs` 後的抽換點；不要把 DB 細節漏進 WebSocket / 遊戲迴圈。

## 資料安全（有真實玩家後尤其重要）

- **絕不**破壞性刪除玩家資料、存檔或既有資料表。
- schema 變更要**可向後相容 / 用 migration**，不要 drop 既有欄位資料。
- **絕不**把密鑰 / 金鑰 / 憑證寫進 repo。機敏值一律走環境變數（見 `.env`，已 gitignore）。
- `data/` 是執行期產生的資料（如玩家建議），已 gitignore，不要提交。

## Git / 發佈

- 在指定的功能分支開發，開 PR 給審查；**不要**對 main 強推。
- commit 訊息清楚、聚焦、繁體中文。
- 完成一個 BACKLOG 項目就在該檔把它打勾，並簡述做了什麼。
- **排程自走營運模式**（見 `docs/AUTONOMOUS_OPS.md`）下例外：小而測試全綠的
  改動可直接進 `main`；**風險大／架構級／動到玩家資料的，仍只開 PR、不自己 merge**。
  上線一律交給維護窗的 `scripts/deploy.sh`，開發迴圈永不自己 `systemctl`。

## 什麼時候要先問人（不要自作主張）

- 任何**破壞性**或**難回復**的操作。
- 任何**對外發佈 / 部署**動作（需要主機 / 機敏設定）。
- 架構級決策（換框架、改資料模型、引入大型相依）。
- 需求有歧義、或某個改動與 GDD 方向相牴觸時。

遇到以上，用提問釐清再動手；其餘照 BACKLOG 自走即可。

## 玩家建議

玩家透過遊戲內建議箱送回饋，存在 `data/suggestions.jsonl`（執行期）。
處理建議時：判斷是否合理、是否符合 GDD 方向、是否小到可直接做；
可直接做的就排進 BACKLOG 或直接實作，需要取捨的先問人。
